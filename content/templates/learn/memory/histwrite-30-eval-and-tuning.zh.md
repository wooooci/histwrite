# Histwrite-30 评测 / 标注 / 调优（让工作流能持续变好）

> 目标：把“你改得舒服/不舒服”的偏好，变成可回归的标准与可学习的数据。
> 使用边界：本文件服务于 **polish/eval/tune**；不应被整体回灌进起草提示词（起草只用 00+01）。

## 1) 评测闭环（每次改稿都能比较）

- 固定一套评分口径（rubric），每一版都用同一套口径打分，避免“今天想改A、明天想改B”。
- 建议至少两类评测：
  - 去 AI 感（线性叙述、句式多样、随机性）
  - 草稿质量（论点集中、学术史对话、证据-解释结合、结构连贯）

输出应落盘（便于回归）：
- 每版得分、关键扣分点、与上版差异。

### 1.1 本项目当前“去 AI 感”硬门槛（只看 1/2/3/4/6）

> 这是“形态学”回归指标：能稳定对比、能做回归，不容易在改稿时跑偏。

- #1 连接词骨架句：`，` 后紧跟单字/短连接（如 `而/但/却/也/并/则`）过密，且分布过均匀。  
- #2 句式模板复用：当…时/不是…而是/……，并……/……，也……/……，却……/越…越…/既…也…/并非…而是… 等在同段/邻段雷同出现。  
- #3 冒号块：概括 + 冒号 + 多句解释/清单式展开。  
- #4 并列名词串：`X、Y与Z` / `X与Y` 的模板化并列。  
- #6 被字句模板：`被`字句密度偏高，或结构反复雷同（“X 被 Y …”循环）。

落地方式（回归最好做“段落级”而非全文级）：
- 评测对象建议固定为 **500–900 字** 的段落/小节输出；否则“模板计数”的绝对阈值会过严。  
- 若必须评测全文：按部分抽样分段评测，并用“比例指标”（如骨架句比例/被字句比例）做主对比。

与 promptfoo 的关系：
- `/histwrite eval run` 会自动在 `promptfooconfig.yaml` 中挂上 `javascript` assert，并写出同目录的 `histwrite_deai_style_assert.cjs`（用于回归 1/2/3/4/6）。  
- 默认阈值可在后续需要时再调；先用同一套阈值跑通“能比较、能回归”的闭环。

精修打法参考：
- 具体“怎么改句法而不碎句化”见 `histwrite-40-deai-polish-playbook.zh.md`。  
- 评分口径以 `.histwrite/learn/rubrics/deai-flow.zh.md` 为准（避免跑偏）。

## 2) 你的逐句修改/批注 = 高价值标注

- 逐句修改与批注可以视为标注的一种（偏好对齐信号强）。
- 颜色约定（若适用）：
  - 蓝色 = 明确要删（后续禁止回潮）
  - 红色 = 你的习惯性新增（提炼为规则后再推全篇）

建议沉淀两类数据：
- rewrite pair：原句 → 你改后的句子（偏好对齐训练数据）
- rule label：你改动背后的原因（例如“减少连接词”“拆长句”“去引号”“去黑话”“少用其”等）

## 3) Prompt 调优优先（先别急着微调）

顺序建议：
1) prompt 约束与流程编排先稳定
2) 回归评测能稳定反映质量提升
3) 标注数据量与偏好稳定后，再考虑微调（TRL/PEFT/LoRA）

## 3.5) judge 与 finalcheck 的分工（这轮必须沉淀）

- `deai-flow.zh.md`：只抓线性叙述与去 AI 形态学问题（1/2/3/4/6），不能拿它替代学术规范校验。
- `draft-quality.zh.md`：看论题、学术史对话、证据—解释结合，也要看门面硬伤（占位符、本地路径、缩写错配、脚注自洽），但不能把 bibliography-style 参考文献误判成脚注失败。
- `finalcheck`：专门抓《历史研究》体例、脚注/参考文献区格式、自洽性与门面问题。
- 经验法则：judge 负责“这篇稿子写得像不像能交导师”，`finalcheck` 负责“这篇稿子的门面会不会一眼露馅”。两者都要跑。

## 4) 工具栈放置位置（怎么嵌进工作流）

- promptfoo：离线 A/B + 回归基准（CI 或本地），用固定样本检查“退化”。
- DSPy / Promptflow：自动找更优 prompt/流程（候选生成 + 评测排序 + 回写 prompt-pack）。
- Langfuse：线上可观测 + prompt 版本与运行日志（谁在用哪个 prompt、产出了什么）。
- Argilla：人工反馈闭环（把“看着不对”的样本变成可学习数据）。
- Trulens / Giskard：运行期守护（更偏质量与风险），RAG 上线后再考虑 RAGAS。

## 5) 不省 token（默认）与工程优化（可选）

- **默认：不省 token，优先一次写对**：允许更长上下文、更多候选、更多证据片段、更多自检（去 AI 回归 + 学术规范回归 + 逻辑自检）。  
- 仍建议以“部分/段落”为单位推进：这不是为了省 token，而是为了减少风格漂移、便于回滚与 A/B 对比评测。  
- 语境交接：每个部分结束产出摘要，下一部分喂“摘要 + 必要证据卡/命中片段”；目的在于保持论证一致与材料可追溯，而非压缩成本。  
- 缓存命中属于工程优化（速度/可复现），不作为写作目标；若希望命中，就保持 system/长期记忆/prompt-pack 稳定。  
- 若未来确实需要控制时延/成本，再启用更激进的切分与摘要压缩，但不能牺牲论证层级与证据链完整性。

## 6) 产物规范（必须可追溯）

- `finalcheck report` 也要归档：尤其是“第一次失败点 → 修完后的清零结果”，这类数据最适合反哺 judge 与 memory。
- 若 judge 对参考文献/档案元数据发生误判，要把“误判触发词 + 正确口径 + 修正后的 rubric 片段”记入长期记忆，而不是只改一次提示词就算了。


- episodes（决策记录）：候选、评分、选择理由
- datasets（训练/回归样本）：原文、改写、标签
- eval report（评测报告）：维度分数、扣分点、版本对比
- prompt-pack（提示词包）：engine/polish/evalRubric 等可版本化配置

## 7) 写完“大修”仍不满意：用评测驱动返工，而不是凭感觉乱改

- 定稿返工要分清两类：
  - **文风/论证返工**：回到 judge、A/B、小样本。
  - **门面/体例返工**：直接看 `finalcheck` 报告逐条修，不要再让 generative judge 模糊给意见。


- 固定“本轮只改 1–2 类问题”，并把它写进 `导出/revise_brief.md`（作为唯一施工单）。  
- 先跑小样本：抽 500–900 字段落做 A/B（同一 rubric，同一阈值），方向对再扩到全文。  
- 返工的每一步都要落盘：revision notes、扣分点清单、以及你最终接受的改法（作为标注样本），然后回写到长期记忆与 style-profile。
