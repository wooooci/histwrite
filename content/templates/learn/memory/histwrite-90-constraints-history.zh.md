### 1.1 “碎句化”的反例提醒（来自用户反馈）

- 用户明确指出：有些紧凑复句本来更好，如果为了“去 AI”拆成 2–3 句短句，反而会让关系松散、学术密度下降。

## 2) “本轮改动（v15→v16）”的可操作化理解（范例）

> 说明：这是一次具体改稿中的问题归因与处理原则，保留于此供复盘参考；不应写入宪法条款。

- 用户明确指出的主要问题：  
  1) 后现代黑话（“重构/嵌入/张力”等）  
  2) “xxx性”名词化过密  
  3) “并”字与连接词密集，且读感均匀像模板  
- 处理原则：  
  - 不做机械替换，而是改造句子承载方式：动词化、拆句（仅必要时）、语序调整、解释落地。  
  - 把黑话词替换为可检验的中文判断句（谁做了什么、导致什么、被谁如何理解）。  
  - 同时保住论文的学术语体底盘，避免变成科普。

- 回归检查（示例指标，用于提醒未来如何量化）：  
  - 高危词残留：如“嵌入/张力/场域/接口/再编码/制度性”等。  
  - “并”字连接过密：尤其“并把/并以”这类模板起手式。  
  - 每次改动后必须做脚注自洽检查（引用号与定义一一对应），防止风格改写破坏学术规范外观。

## 3) 本轮（v24→v28）高价值经验：要沉进系统，而不是停留在一次修稿里

### 3.1 门面校验必须独立成关卡

- 事实证明：单靠 generative judge 很容易只盯文风，不会稳定抓住占位符、缩写错配、本地路径、脚注/参考文献不自洽这些“门面硬伤”。
- 结论：定稿前必须把 `finalcheck` 当成单独关卡；judge 与 `finalcheck` 不能互相替代。

### 3.2 脚注区与参考文献区不能混判

- 《历史研究》体例下，正文脚注与文末参考文献的外形本来就不一样，尤其英文材料更明显。
- 结论：以后凡是做 judge / lint / finalcheck，都要先区分“正文脚注问题”与“参考文献区问题”；不能因为 bibliography-style 写法不像脚注，就误报。

### 3.3 英文材料的“像元数据”不等于假

- `Repository / Collection / Box / Folder / Item No. / Gale / RFE / reprinted from` 之类字段，在英文档案、数据库材料、重印刊物里经常就是有效出处链。
- 结论：judge 以后看到这类字段，应先判断它是不是来源链的一部分，而不是把它们当成“杂乱元数据”直接扣分。

### 3.4 占位符与缩写错配属于高优先级硬伤

- `**###**`、未定义或未使用脚注、同一缩写映射到不同中文名，这些问题会立刻破坏可信度，而且很容易被导师一眼看出。
- 结论：这类问题要在任何文风精修之前先清干净。

### 3.5 反哺方式：改 rubric，不只改一次稿子

- 一轮修稿里真正有价值的，不只是稿子变干净了，而是把“误判模式—正确口径—新规则”沉到 memory 与 rubrics。
- 结论：以后只要出现新的稳定误判（例如 bibliography-style 误报、把列举一律改成动作补丁），都要优先改系统文件，再去改稿。
